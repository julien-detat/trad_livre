DÉCLARATION HTML et CSS sont tout les deux des exemples de langages déclaratifs. Un auteur écrivant dans un language déclaratif décrit un résultat souhaité sans fournir les instructions étape par étape à l’ordinateur qui traite le fichier. Avec l’HTML, vous pouvez décrire la nature du contenu, des paragraphes, des titres, des champs de formulaire, etc; sans avoir à expliquer exactement ce que le navigateur doit faire avec ces informations. Le CSS quant à lui, vous permet de décrire l’apparence désirée du contenu; les couleurs, les bordures, etc; sans avoir à écrire un programme pour appliquer ces styles. La plupart des langages ne sont pas déclaratifs, ils sont impératifs. Perl, Java, C++… Sont autant d’exemples de langages impératifs. Si vous écrivez dans l’un de ces langages, vous devez fournir des instructions précises à l’ordinateur qui interprétera votre code. Si les langages impératifs vous donnent plus de puissance et de précision que les langages déclaratifs, cet avantage a un prix. Les langages impératifs ont tendance à être plus difficiles à apprendre que les langages déclaratifs. C’est aussi plus compliqué d’y appliquer la Loi de Postel.Une seule erreur; virgule ou un point-virgule mal placé; et c’est le programme entier qui peut échouer. Si, Un élément mal orthographié en HTML ou une accolade manquante en CSS peut causer des maux de tête, les programmes impératifs quant à eux doivent absolument être bien formés ou ils ne fonctionneront pas du tout. Les langages impératifs tels que PHP, Ruby, et Python sont les langages communément utilisé sur le World Wide Web, lisant et écrivant les enregistrements des bases de données, traitant des entrées, et éxécutant des algorithmes complexes. Dès lors que vous écrivez du côté du serveur, vous pouvez choisir le langage de programmation. Si vous ne pouvez pas connaître l’identité de l’utilisateur final du navigateur, vous pouvez contrôler les capacités de votre serveur. Si vous voulez utiliser un code impératif fonctionnant sur un navigateur web, Javascript est votre unique option. SCRIPTAGE L’idée d’exécuter un programme depuis l’intérieur d’une page web est aussi vieille que le web lui-même. Voici un e-mail provenant d’une discussion sur le WWW: « Je voudrais savoir, si oui ou non quelqu’un a élargi WWW de façon qu’il serait possible de démarrer des programmes arbitrairement en appuyant un bouton dans le navigateur web. » Auquel Tim Berners-Lee, le créateur du World Wide Web, a répondu : « Très bonne question. Le problème est le langage de programmation. Vous avez besoin de quelque chose de vraiment puissant, mais en même temps omniprésent. Rappelez vous qu’une facette du web est un lectorat universel. Or il n’existe pas de langage de programmation universel.» C’était en 1992. 4 ans plus tard, le premier langage de programmation universel apparu en 1996. Il fut écrit en 10 jours par un programmeur de Netscape nommé Brendan Eich. Le langage a changé plusieurs fois de noms. D’abord appelé Mocha, puis officiellement lancé sous le nom de LiveScript. Finalement, le département marketing décida de le renommer Javascript, en espérant que le nom surferait sur la vague de battage médiatique provoquée par le nouveau langage Java. Mais, ces langages n’ont rien en commun. Java est à JavaScript comme jambon est à hamster. LES MODÈLES DE PROGRÈS JavaScript a permis aux web designers de modifier les pages web après leur publication. Rapidement, deux façons de procédé sont apparues : les renversements et la validation de formulaire. Si aujourd’hui afficher des images lors du suvol d’un lien peut sembler être un usage impertinent d’un langage de programmation, dans les années 90, il n’y avait pas d’autre moyen de créer un effet de survol. Avant Javascript, un formulaire devait être envoyé à un serveur web pour s’assurer que tout les champs obligatoires étaient remplis, ou que les informations entrées correspondaient au format demandé. C’est encore le cas aujourd’hui, mais nous n’avons plus besoin de JavaScript. On peut créer des effets de renversements en utilisant la pseudo-classe css : hover. On peut aussi valider les champs de formulaire en utilisant les éléments html REQUIRED et TYPE. Encore et encore, ce schéma se reproduit : une solution est créée dans un langage impératif et si elle devient assez populaire, ce n’est qu’une question de temps avant de la voir migrer vers un langage déclaratif. Quand une fonctionnalité est invalide dans un langage déclaratif, c’est non seulement facile à écrire, c’est aussi plus robuste. La haute tolérance aux erreurs HTML et CSS signifie que beaucoup d’erreurs de création ou problèmes de support du navigateur sont laissées de côté; le navigateur ignore simplement ce qu’il ne comprend pas et poursuit son analyse. C’est souvent assez bon. En revanche, si vous donnez à un navigateur un code JavaScript mal formé ou tentez d’utiliser un élément qu’il ne supporte pas, non seulement le navigateur affichera une erreur, mais il s’arrêtera d’analyser le script et refusera d’aller plus loin. UNFORGIVEN (La tolérance d’html aux erreurs) La gestion des erreurs d'HTML lui a permis de gagner en puissance avec le temps. Étonnamment, il y a eu une tentative de supprimer cette superpuissance du HTML. Après la standardisation du HTML version 4 en 1999, le World Wide Web Consortium a publié XHTML 1.0. Cela reformule HTML en fonction des règles du format de données XML. Alors que HTML peut avoir des noms et des attributs de balises en majuscules ou en minuscules, XML exige qu’ils soient tous en minuscules. Il y avait d’autres différences: tous les attributs devaient être cités, et les éléments autonomes comme IMG ou BR nécessitaient une barre oblique de fermeture. XHTML 1.0 n’a pas ajouté de nouvelles fonctionnalités au langage. C’était simplement une manière plus stricte d’écrire le balisage. XHTML 2.0 était une proposition différente. Cela supprimerait non seulement les éléments établis IMG, mais implémenterait également le modèle draconien de traitement des erreurs de XML. S’il existe une seule erreur dans le document XML (un attribut non cité ou une barre de fermeture manquante), alors l’analyseur doit s’arrêter immédiatement et refuser de rendre quoi que ce soit. XHTML 2.0 est mort sur le fil. Sa pureté théorique a été catégoriquement rejetée par les créateurs de sites Web. Les designers de sites Web ont à juste titre refusé de publier un format qui échouerait complètement au lieu d’essayer de remédier à une erreur. Étrange alors, que des années plus tard, les concepteurs Web créaient avec joie des sites Web entiers en utilisant JavaScript, un langage qui partage le modèle de traitement d’erreurs impitoyable du XML. Ils ne les appelaient pas des sites Web. Ils les appelaient des applications Web. Cette distinction réconfortait très mal quelqu’un, qui ne pouvait pas accomplir sa tâche parce qu’un service comptait sur JavaScript pour une fonctionnalité cruciale. Malgré le modèle fragile de traitement des erreurs de JavaScript, les concepteurs Web (deviennent) sont de plus en plus dépendants de JavaScript au fil du temps.
